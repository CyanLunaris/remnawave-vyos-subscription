# xray-core Kernel Selection — Design Spec

**Date:** 2026-03-29
**Status:** Approved
**Scope:** Add xray-core as an alternative proxy kernel to remnawave-vyos-subscription, selectable via `config.env` and the TUI.

---

## Problem

sing-box XTLS Reality performance and stability is inferior to xray-core, which is the original author of the XTLS/Vision/Reality stack. Users want to switch kernels without changing VyOS configuration.

---

## Approach: tun2socks + xray

Keep `tun0` and VyOS policy routing **unchanged**. Add `tun2socks` as a thin converter between the existing TUN interface and xray's SOCKS5 inbound.

```
[sing-box mode]  tun0 → sing-box (TUN inbound + VLESS/XTLS outbound)
[xray mode]      tun0 → tun2socks → xray (SOCKS5 inbound + VLESS/XTLS outbound)
```

No VyOS config changes. Kernel choice is purely a container concern.

---

## Kernel Selection

`config.env` gains one new variable:

```
PROXY_KERNEL=singbox   # default; or: xray
```

**Single source of truth:** `config.env` is the sole persistent store for kernel selection. `state_manager.py` does not track `PROXY_KERNEL`. On container restart, `daemon.py` reads `PROXY_KERNEL` from `config.env` and starts the correct kernel.

---

## Binary Sources

All binaries downloaded only if absent (idempotent). Architecture detected via the existing `_get_arch()` helper.

| Binary | GitHub repo | Asset (amd64) | Asset (arm64) |
|---|---|---|---|
| `sing-box` | `SagerNet/sing-box` | `sing-box-{ver}-linux-amd64.tar.gz` | `sing-box-{ver}-linux-arm64.tar.gz` |
| `xray` | `XTLS/Xray-core` | `Xray-linux-64.zip` | `Xray-linux-arm64-v8a.zip` |
| `tun2socks` | `xjasonlyu/tun2socks` | `tun2socks-linux-amd64.zip` | `tun2socks-linux-arm64.zip` |

Both xray and tun2socks archives contain a single binary at their root. Extract with `zipfile` (same pattern as existing `_download_sing_box` using `tarfile`).

**Version stored at download time:** when `ensure_xray()` downloads xray, it runs `xray version`, parses the first line (e.g., `Xray 1.8.24 ...`), and writes the version string to `/etc/xray/.version`. Status bar reads from this file — no subprocess call at display time.

**Dockerfile must not set `XRAY_LOCATION_ASSET`** — the project sets it explicitly in the xray subprocess environment.

## Geo Assets

xray and sing-box use **incompatible geo formats** stored in **separate directories**.

| File | Format | Path | Source |
|---|---|---|---|
| `geoip-ru.srs` | sing-box binary rule-set | `/etc/sing-box/` | `SagerNet/sing-geoip` |
| `geosite-category-ru.srs` | sing-box binary rule-set | `/etc/sing-box/` | `SagerNet/sing-geosite` |
| `geoip.dat` | v2fly protobuf | `/etc/xray/` | `v2fly/geoip` → `geoip.dat` |
| `geosite.dat` | v2fly protobuf | `/etc/xray/` | `v2fly/domain-list-community` → `dlc.dat` (renamed) |

`XRAY_LOCATION_ASSET=/etc/xray/` is set in the xray subprocess env (parent env inherited, this key overridden). The `/etc/sing-box/` directory is never referenced by xray.

**Atomic downloads:** all geo files are downloaded to a `.tmp` file in the same directory and renamed on success. On any download failure the `.tmp` file is deleted. This prevents partial-update corruption on retry.

---

## Routing Rules

The same `config.env` vars drive routing for both kernels:

```
GEO_DIRECT_IP=private,ru
GEO_DIRECT_SITE=category-ru
```

- sing-box: maps to `.srs` `rule_set` entries (existing behaviour)
- xray: maps to `geoip:{code}` / `geosite:{code}` routing rules

Both kernels produce identical traffic policy for the same config values.

## xray Config Structure

Generated by `xray_config_generator.py`. Reuses the existing `ConfigSettings` dataclass.

```json
{
  "log": {
    "loglevel": "warning",
    "error": "/var/log/remnaproxy/xray-error.log"
  },
  "inbounds": [{
    "tag": "socks-in",
    "protocol": "socks",
    "listen": "127.0.0.1",
    "port": 7891,
    "settings": { "udp": true }
  }],
  "outbounds": [
    { "tag": "proxy",  "protocol": "...", "settings": { ... } },
    { "tag": "direct", "protocol": "freedom" }
  ],
  "routing": {
    "domainStrategy": "IPIfNonMatch",
    "rules": [
      { "type": "field", "ip":     ["private", "geoip:ru"],  "outboundTag": "direct" },
      { "type": "field", "domain": ["geosite:category-ru"],  "outboundTag": "direct" }
    ]
  }
}
```

### ParsedNode → xray outbound field mapping

| ParsedNode field | xray VLESS field | Notes |
|---|---|---|
| `host` | `vnext[0].address` | |
| `port` | `vnext[0].port` | |
| `uuid` | `vnext[0].users[0].id` | |
| `flow` | `vnext[0].users[0].flow` | `xtls-rprx-vision` for XTLS |
| `network` | `streamSettings.network` | tcp / ws / grpc |
| `security` | `streamSettings.security` | `tls` or `reality` |
| `sni` | `streamSettings.tlsSettings.serverName` | |
| `fingerprint` | `streamSettings.tlsSettings.fingerprint` | via `utlsSettings` |
| `reality_pbk` | `streamSettings.realitySettings.publicKey` | |
| `reality_sid` | `streamSettings.realitySettings.shortId` | |
| `ws_path` | `streamSettings.wsSettings.path` | |
| `ws_host` | `streamSettings.wsSettings.headers.Host` | |
| `grpc_service` | `streamSettings.grpcSettings.serviceName` | |

VMess, Trojan, Shadowsocks follow their respective xray outbound schemas.
**xhttp transport is not supported** in xray-core — config generation raises `ValueError` with a clear message. No silent fallback.

## XrayRunner

### Subprocess monitoring

`XrayRunner._watch()` monitors **both** subprocesses (xray and tun2socks) using two threads. Either subprocess exiting unexpectedly is treated as a crash event and triggers a restart of **both**. This ensures tun2socks crashes (which do not affect the SOCKS5 probe) are detected and acted upon.

### Startup sequence

```
XrayRunner.start():
  1. Start xray subprocess
     (env = os.environ | {"XRAY_LOCATION_ASSET": "/etc/xray/"})
  2. TCP probe 127.0.0.1:7891 every 200ms, timeout 10s
     → RuntimeError("xray SOCKS5 not ready") on timeout
  3. Start tun2socks:
     tun2socks -device tun://tun0 -proxy socks5://127.0.0.1:7891 -loglevel warning
     (tun2socks stderr redirected to same log file as xray for unified log_path)
  4. Start _watch() threads for both subprocesses
  5. Log "XrayRunner started"
```

**tun2socks device format:** `tun://tun0` (URL form required by xjasonlyu/tun2socks v2+).
**tun2socks stderr** is redirected to `/var/log/remnaproxy/xray-error.log` (same file as xray's log), so `runner.log_path` covers both processes — no separate log plumbing in TUI.

### Log interface

Both `SingboxRunner` and `XrayRunner` expose:

```python
runner.log_path: str  # absolute path to active log file
```

TUI reads from `runner.log_path` with no conditional branching.

## Restart Behaviour

**Sliding window:** `MAX_RESTARTS = 5` within 60 seconds. On each crash, timestamps of all crashes are recorded. Before deciding to stop, discard timestamps older than 60 seconds and count the remainder. If count ≥ `MAX_RESTARTS`, mark runner as failed. This prevents unbounded crash accumulation while allowing long-running processes to recover from occasional failures.

Applied to both `SingboxRunner` (pre-existing gap fixed) and `XrayRunner`.

On crash restart: the full `XrayRunner.start()` sequence (including SOCKS5 probe) is repeated.

---

## `stop()` Contract

`stop()` is **always non-raising**. Internally:

1. Send SIGTERM to all managed subprocesses
2. Wait up to 5 seconds for each to exit
3. Send SIGKILL to any still alive
4. Swallow all exceptions

Callers can assume all subprocesses are dead after `stop()` returns.

---

## `daemon.restart_kernel()` — Hot Switch

Stop-then-start. `tun0` preserved across switch.

```
restart_kernel(new_kernel, log_callback=None):
  1. old_kernel = read PROXY_KERNEL from config.env
  2. Write new PROXY_KERNEL to config.env
  3. current_runner.stop()           # always non-raising; tun0 stays up
  4. sync_main(log_callback=log_callback)  # download + generate config
  5. new_runner = make_runner(new_kernel)
  6. new_runner.start()              # includes SOCKS5 probe for xray
  7. On exception at step 4, 5, or 6:
       Write old_kernel back to config.env
       try: make_runner(old_kernel).start()
       except: log critical — both kernels failed
       raise original exception
  8. self._runner = new_runner
```

**tun0 during switch:** packets arriving between step 3 and step 6 are dropped by the OS (no reader). Brief interruption is acceptable and communicated in the TUI modal.

---

## sync.py Changes

`sync_main()` gains an optional `log_callback=None` parameter (existing callers unaffected).

Internally:
1. Reads `PROXY_KERNEL` from env
2. Calls appropriate `ensure_*` functions and config generator
3. Writes config to `xray-config.json` (`PROXY_KERNEL=xray`) or `config.json` (`singbox`)
4. Emits progress strings via `log_callback(line)` if provided
5. Does **not** reload the running process

## TUI Changes

### Settings tab — kernel toggle

```
Proxy Kernel   [ sing-box ]  [ xray ]
```

Active kernel highlighted. On click:

1. Open non-closeable progress modal
2. Call `daemon.restart_kernel(new_kernel, log_callback=modal.append_line)` in background thread
3. On success: close modal, update status bar
4. On failure: close modal with error message; `config.env` already reverted by `restart_kernel()`

### Status bar

```
Kernel    xray v1.8.24
```

Version read from `/etc/xray/.version` (written at download time by `ensure_xray()`). Falls back to `xray v?.?.?` if file absent.

## Error Handling Summary

| Scenario | Behaviour |
|---|---|
| Download fails (partial) | `.tmp` file deleted; `restart_kernel()` raises; reverts `config.env`; restarts old kernel |
| xray SOCKS5 not ready in 10s | `start()` raises; same revert path |
| Either subprocess crashes | `_watch()` detects either exit; restarts both; counts toward sliding window |
| 5 crashes within 60s | Runner marked failed; TUI shows `Kernel: xray ✗`; user must switch manually |
| xhttp node in xray mode | Config generation raises `ValueError`; sync falls back to cached config |
| `stop()` hangs | SIGKILL after 5s; `stop()` always returns |
| Both kernels fail on switch | Critical log; runner left in failed state; user intervention required |

## Testing

New test files following existing pytest style. Network calls mocked via `unittest.mock.patch`.

**`tests/test_xray_config_generator.py`** (new):
- Config structure for each protocol (vless/vmess/trojan/ss)
- XTLS flow field present for vless+reality nodes
- Routing rules reflect `GEO_DIRECT_IP` / `GEO_DIRECT_SITE` config values
- xhttp node raises `ValueError`

**`tests/test_binary_manager.py`** (extend):
- `ensure_xray()` with mocked GitHub API + download; version file written
- `ensure_tun2socks()` with mocked download
- `ensure_xray_rule_sets()` downloads to `/etc/xray/`, atomically; missing files only
- Architecture mapping: amd64 → `Xray-linux-64.zip`, arm64 → `Xray-linux-arm64-v8a.zip`

**`tests/test_daemon.py`** (extend):
- `Daemon` selects `XrayRunner` when `PROXY_KERNEL=xray`
- `Daemon` selects `SingboxRunner` when `PROXY_KERNEL=singbox` or unset
- `restart_kernel()` reverts `config.env` and restarts old runner on failure
- `stop()` is non-raising even when subprocess is already dead
- Sliding window restart counter: 5 crashes within 60s → failed; 5 spread over >60s → continues

Existing sing-box tests **untouched**.
