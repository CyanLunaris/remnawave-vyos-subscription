# xray-core Kernel Selection — Design Spec

**Date:** 2026-03-29
**Status:** Approved
**Scope:** Add xray-core as an alternative proxy kernel to remnawave-vyos-subscription, selectable via `config.env` and the TUI.

---

## Problem

sing-box XTLS Reality performance and stability is inferior to xray-core, which is the original author of the XTLS/Vision/Reality stack. Users want to switch kernels without changing VyOS configuration.

---

## Approach: tun2socks + xray

Keep `tun0` and VyOS policy routing **unchanged**. Add `tun2socks` as a thin converter between the existing TUN interface and xray's SOCKS5 inbound.

```
[sing-box mode]  tun0 → sing-box (TUN inbound + VLESS/XTLS outbound)
[xray mode]      tun0 → tun2socks → xray (SOCKS5 inbound + VLESS/XTLS outbound)
```

No VyOS config changes. Kernel choice is purely a container concern.

---

## Architecture

### Kernel selection

`config.env` gains one new variable:

```
PROXY_KERNEL=singbox   # default; or: xray
```

`daemon.py` reads this on startup and instantiates the matching runner. `sync.py` reads the same variable to call the correct config generator and binary manager.

### Component map

```
daemon.py
 ├── SingboxRunner      # existing behaviour, untouched
 └── XrayRunner
      ├── tun2socks subprocess   (tun0 → socks5://127.0.0.1:7891)
      └── xray subprocess        (socks5 inbound + outbound + routing)

binary_manager.py
 ├── ensure_sing_box()           # existing
 ├── ensure_xray()               # new — GitHub: XTLS/Xray-core
 ├── ensure_tun2socks()          # new — GitHub: xjasonlyu/tun2socks
 └── ensure_xray_rule_sets()     # new — downloads geoip.dat + geosite.dat

xray_config_generator.py         # new file, mirrors config_generator.py
 └── generate_xray_config(node, settings) -> dict
```

## xray Config Structure

Generated by `xray_config_generator.py`:

```json
{
  "log": { "loglevel": "warning", "access": "/var/log/remnaproxy/xray-access.log", "error": "/var/log/remnaproxy/xray-error.log" },
  "inbounds": [{
    "protocol": "socks",
    "listen": "127.0.0.1",
    "port": 7891,
    "settings": { "udp": true }
  }],
  "outbounds": [
    { "tag": "proxy",  "protocol": "vless|vmess|trojan|shadowsocks", "..." },
    { "tag": "direct", "protocol": "freedom" }
  ],
  "routing": {
    "domainStrategy": "IPIfNonMatch",
    "rules": [
      { "type": "field", "ip":     ["private", "geoip:ru"],   "outboundTag": "direct" },
      { "type": "field", "domain": ["geosite:category-ru"],   "outboundTag": "direct" }
    ]
  }
}
```

`XRAY_LOCATION_ASSET` env var points xray at `/etc/sing-box/` so it finds `geoip.dat` and `geosite.dat` alongside existing sing-box rule-sets.

Supported protocols: **vless** (with XTLS Reality / Vision flow), **vmess**, **trojan**, **shadowsocks** — same set as sing-box path, parsed from the same `ParsedNode`.

## Geo Files

| File | Source | Used by |
|---|---|---|
| `geoip-ru.srs` | SagerNet/sing-geoip | sing-box |
| `geosite-category-ru.srs` | SagerNet/sing-geosite | sing-box |
| `geoip.dat` | v2fly/geoip (latest release) | xray |
| `geosite.dat` | v2fly/domain-list-community `dlc.dat` (latest) | xray |

All four files live in `rule_set_dir` (`/etc/sing-box/`). Each is downloaded only if absent.

## Daemon Changes

### `XrayRunner`

Mirrors `SingboxRunner` structure:

- `start()` — launches tun2socks then xray as `subprocess.Popen`
- `stop()` — terminates both processes gracefully (SIGTERM → wait 10s → SIGKILL)
- `_watch()` — background thread; if either subprocess exits unexpectedly, restarts **both** (tun2socks + xray are coupled — tun2socks alone is useless without xray)
- Restart cap: same `MAX_RESTARTS = 5` as sing-box runner

### `daemon.restart_kernel()`

New public method used by TUI for hot-switching:

1. `current_runner.stop()`
2. `sync_main()` — re-generates config + downloads any missing binaries
3. Yield log lines to caller via callback (for TUI progress display)
4. Instantiate and `start()` new runner
5. Return success/failure

## TUI Changes

### Kernel switcher (Settings tab)

Two toggle buttons: **sing-box** / **xray**. Active kernel highlighted.

On click:
1. Write new `PROXY_KERNEL` value to `config.env`
2. Open progress modal — streams `daemon.log` lines in real time
3. Call `daemon.restart_kernel()` in background thread
4. On success: close modal, update status bar → `Kernel: xray v1.x.x`
5. On failure: close modal with error, revert `config.env` to previous value, old kernel stays running

### Status bar addition

Existing status panel gains one row:

```
Kernel    xray v1.8.24
```

Version string obtained by running `<binary> version` and parsing first line.

---

## Error Handling

| Scenario | Behaviour |
|---|---|
| Download fails (no network) | Log error, revert `PROXY_KERNEL`, old kernel keeps running |
| xray fails to start | `XrayRunner` logs stderr, marks kernel as failed, TUI shows `Kernel: xray ✗` |
| tun2socks crashes mid-session | `XrayRunner._watch()` detects exit, restarts both processes |
| Kernel switch during active traffic | Graceful stop then start — brief connection interruption is acceptable |
| Binary already present | `ensure_*` functions skip download (idempotent) |

## Testing

New test files following existing pytest style (no real network calls — mock urllib):

- `tests/test_xray_config_generator.py`
  - Config structure for each protocol (vless/vmess/trojan/ss)
  - Routing rules presence: private + geoip:ru + geosite:category-ru → direct
  - XTLS flow field present for vless+reality nodes

- `tests/test_binary_manager.py` (extend)
  - `ensure_xray()` with mocked GitHub API + HTTP download
  - `ensure_tun2socks()` with mocked download
  - `ensure_xray_rule_sets()` downloads only missing files

- `tests/test_daemon.py` (extend)
  - `Daemon` selects `XrayRunner` when `PROXY_KERNEL=xray`
  - `Daemon` selects `SingboxRunner` when `PROXY_KERNEL=singbox` or unset

Existing tests for sing-box path remain **untouched**.
